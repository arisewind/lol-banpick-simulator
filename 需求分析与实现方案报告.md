## 1. 待实现需求：数据分析功能（推荐 BP / 协同度 / 对阵优势）

### 1.1 TODO 原始描述与隐含前提

TODO 写的是"集成第三方 API，实现推荐系统"。这隐含一个前提：**推荐数据必须来自外部 API**。核查后这个前提不成立或代价过高，是整个需求被搁置的真正原因。下面先拆解数据源，再给方案。

### 1.2 数据源可行性核查

| 数据源 | 能提供什么 | 能否驱动"推荐/胜率/克制" | 代价/风险 |
|--------|-----------|------------------------|----------|
| **Data Dragon**（项目已用） | 英雄静态属性：标签（Mage/Tank…）、基础数值、技能文本 | ❌ 仅静态元数据，无胜率、无对线关系 | 免费、无 key、已在用 |
| **Riot Games API** | 比赛记录、召唤师数据、排位信息 | ⚠️ **不直接提供胜率/克制**，只给原始对局，需自己聚合百万局才能算出有意义胜率 | 需 API key（生产要申请、有速率限制、个人 key 禁商用） |
| **U.GG / OP.GG / LoLalytics** | 现成胜率/克制/出装 | ✅ 数据最直接 | ❌ 无官方 API，靠爬虫，违反 TOS，封禁风险，格式随时变 |
| **社区静态克制表**（人工维护） | tag 级克制、经典 synergy 组合 | ✅ 可做规则引擎 | 维护成本，但可控、可测 |

**结论：** 没有一个免费、稳定、合规的公开 API 能直接给出"推荐 + 胜率 + 克制"。把需求绑死在"第三方 API"上是它被搁置的根因——**不是技术上做不了，而是数据源选型走不通**。

### 1.3 三个实现方案

**方案 A —— 本地规则引擎（推荐，MVP）**
- 用英雄 `tags` + 人工维护的两张表（`synergy.json` / `counter.json`）做纯本地计算：
  - **协同度**：队伍标签多样性 + 命中 synergy 表的组合加分。
  - **对阵优势**：按 tag 克制规则（Assassin→Marksman/Mage、Tank→Assassin、Support 互补等）+ counter 表逐对求差。
  - **推荐**：基于当前双方已 pick，推荐"补齐己方缺位标签"或"克制对方核心"的英雄，过滤掉已 ban/pick。
- 优点：完全离线、即时、纯函数易测试、不增加运行时依赖、无 key/合规风险。
- 代价：精度取决于规则表质量，非真实赛事数据。

**方案 B —— 用户自导入数据**
- 借助需求 2 的导入导出能力，允许用户导入一份 `{heroId: {winRate, counters:[]}}` 的 JSON，本地查询。
- 优点：数据可更新、来源灵活。
- 代价：需要用户自行获取数据，门槛高。

**方案 C —— 对接 Riot API + 离线聚合**
- 申请 Riot key，定期拉取并聚合胜率，落库。
- 优点：数据真实。
- 代价：基础设施重（聚合管道 + 存储 + key 管理），不适合当前单机 Electron 工具的规模。

### 1.4 推荐落地路径

**采用方案 A**，把 [DataContext.tsx](src/contexts/DataContext.tsx) 的三个 TODO 替换为对纯函数的调用：

```
src/
├── analysis/                      ← 新增
│   ├── rules/
│   │   ├── synergy.ts             # 纯函数：阵容协同打分
│   │   ├── matchup.ts             # 纯函数：对阵优势计算
│   │   └── recommend.ts           # 纯函数：推荐列表
│   └── data/
│       ├── synergy.json           # 人工维护的 synergy 表
│       └── counter.json           # 人工维护的克制表
└── contexts/DataContext.tsx       # analyze() 调用上述纯函数
```

> **关键收益：纯函数可被单元测试覆盖**，正好契合后续测试任务（模仿 [AGENTS-Subject of Imitation.md](AGENTS-Subject%20of%20Imitation.md) 的"优先提取纯函数测试"范式）。

## 2. 待实现需求：文件导入导出（保存/加载 BP 记录）

### 2.1 可行性：**高**，无外部依赖阻塞

Electron 主进程具备完整 Node 能力（`fs`、`dialog`），preload 已预留通道，缺的只是三段样板代码 + 一个 BP 状态恢复入口。

### 2.2 实现方案

**(1) 主进程 handler**（新增 `src/main/ipc/dataHandler.js`，仿 [heroHandler.js](src/main/ipc/heroHandler.js) 范式）：

```js
ipcMain.handle('export-data', async (_event, data) => {
  const { canceled, filePath } = await dialog.showSaveDialog({
    defaultPath: `bp-record-${Date.now()}.json`,
    filters: [{ name: 'BP Record', extensions: ['json'] }],
  })
  if (canceled || !filePath) return { success: false, canceled: true }
  fs.writeFileSync(filePath, JSON.stringify({ version: 1, ...data }, null, 2), 'utf-8')
  return { success: true, filePath }
})

ipcMain.handle('import-data', async () => {
  const { canceled, filePaths } = await dialog.showOpenDialog({
    properties: ['openFile'], filters: [{ name: 'BP Record', extensions: ['json'] }] })
  if (canceled || !filePaths.length) return { success: false, canceled: true }
  const raw = fs.readFileSync(filePaths[0], 'utf-8')
  return { success: true, data: JSON.parse(raw) }
})
```
并在 [main.js:46](src/main/main.js#L46) 旁注册 `registerDataHandlers()`。

**(2) BPContext 新增恢复入口**（[BPContext.tsx](src/contexts/BPContext.tsx) 缺这个能力）：

```ts
loadSnapshot: (snap: { currentPhase: number; blueTeam: TeamState;
                       redTeam: TeamState; history: HeroAction[] }) => void
```
内部直接 `setState(snap)`。导入时先校验 schema，再调用。

**(3) 接线**：[App.tsx:64-70](src/App.tsx#L64-L70) 的导出/导入按钮接上：
- 导出：`window.electronAPI.exportData({ currentPhase, blueTeam, redTeam, history })`
- 导入：`const { data } = await window.electronAPI.importData(); loadSnapshot(data)`

**(4) 数据格式**（版本化，便于后续迁移）：
```json
{ "version": 1, "currentPhase": 12, "blueTeam": {...}, "redTeam": {...}, "history": [...], "createdAt": "..." }
```

### 2.3 风险点
- 导入需做 **schema 校验 + 字段白名单**，避免恶意/损坏 JSON 破坏状态（防御性）。
- `Date.now()` 时间戳需在 renderer 侧生成（导出按钮点击时），主进程不应信任传入的时间。

## 3. 搁置功能再评估

TODO 把 **"推荐 BP"** 和 **"快速统计"** 放进"暂不实现"，理由是：

> 依赖尚未实现的分析逻辑，DataContext 目前为 stub，暂不具备实现条件，先搁置。

### 3.1 再评估结论：**搁置可以解除**

- 依赖关系本身成立——它们确实要靠 `analyze()`。
- 但搁置的**隐含前提**是"分析逻辑必须接第三方 API，而 API 走不通 ⇒ 分析做不了 ⇒ 推荐/统计做不了"。
- §1.2 已论证：分析逻辑**不必依赖第三方 API**，用本地规则引擎（方案 A）即可实现 MVP。
- 因此推荐 BP 与快速统计**不是独立需求**，它们就是"数据分析功能"的输出展示层：
  - 推荐 BP = `analysis/rules/recommend.ts` 的结果 → `recommendations`
  - 快速统计 = `synergy.ts` + `matchup.ts` 的结果 → `synergyAnalysis` / `matchupAnalysis`
- **DataContext 的 state 结构（recommendations/synergyAnalysis/matchupAnalysis）和 [AnalysisPanel.tsx](src/components/analysis/AnalysisPanel.tsx) 的 UI 都已就位**，只差把 stub 换成真实计算。

### 3.2 建议
把 TODO 的"待实现-数据分析功能"与"搁置-推荐 BP/快速统计"**合并为单一需求「数据分析模块」**，按方案 A 落地，搁置状态随之消除。
