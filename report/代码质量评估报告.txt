4. 架构问题 ⚠️ 整体良好但有优化空间
Context 使用 - 规范：


┌─────────────────────────────────────────────────────┐
│ HeroContext → BPContext → DataContext (嵌套层级)   │
└─────────────────────────────────────────────────────┘
优势：

Context 职责清晰：英雄数据、BP 流程、分析数据分离
自定义 Hook 命名规范：useBP、useHeroes、useData
Context 值包含完整状态 + 操作方法
潜在性能问题：

HeroContext 性能优化良好:


const filteredHeroes = useMemo(() => { ... }, [state.heroes, state.searchQuery, state.selectedTags])
已使用 useMemo 优化计算密集的过滤操作
BPContext 性能风险:


const getCurrentPhase = useCallback((): BPPhase | null => {
  if (state.currentPhase >= BP_PHASES.length) return null
  return BP_PHASES[state.currentPhase]
}, [state.currentPhase])
依赖项每次 state 变化都会更新，可能导致消费者重渲染
建议: 改为 useMemo 或 memoize BP_PHASES[state.currentPhase]
无 Context 分割优化:

所有消费者订阅整个 Context value
大型应用可考虑拆分为多个 Context（状态-only vs 操作-only）
状态管理清晰度 - 良好：

状态流向清晰: Context → 自定义 Hook → 组件
本地状态使用合理（如 AnalysisDrawer 的 open、HeroGrid 的 showTags）
无 prop drilling 问题
